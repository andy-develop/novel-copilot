# 长篇小说写作助手 (Novel Copilot) v4.0

超长篇小说（30万字+）的核心难题：**写到后面忘了前面**。本 skill 通过结构化状态追踪 + 逻辑校验 + 剧情发散 + **全自动写作引擎** + **叙事纪律引擎** + **v4.0 智能增强引擎**（角色深度/伏笔发现/质量评分/章节解剖/情感编排/对话潜台词/章节变速/效率优化），确保长篇不崩盘、不注水、不让读者弃书。

## 核心理念

> 写作是发散的，校验是收敛的。先发散，再校验，最后落笔。

**叙事三优先级**（冲突时高优先级不可妥协）：
- **P0 = 节奏 + 爽度 + 画面感** — 读者留不留得住、爽不爽、记不记得住，全靠这三条
- **P1 = 一致性 + 散文质量** — 不崩盘、不啰嗦
- **P2 = 大纲 + 字数 + 状态文件** — 工具而非枷锁

> 当 P0 与 P2 冲突时（如大纲要求铺垫但铺垫会拖慢节奏），**节奏爽感优先**，调整大纲而非牺牲节奏。

### 模块一：角色状态库 (characters.yaml)
追踪角色当前状态，每次写到时自动校验。字段：`id, name, aliases, status(alive|dead|missing|sealed), death_chapter, death_cause, location, abilities, relationships, inventory, notes, last_appeared`。校验：死者无行动、物品唯一持有、能力增减有事件支撑、位置移动合理。

### 模块二：世界设定库 (world.yaml)
追踪地理、势力、能力体系。字段：`geography(id,name,type,location,status,destroyed_chapter), factions, power_system(name,ranks,rules)`。校验：已毁地点不再引用、势力变动有事件、能力使用不违反规则。

### 模块三：剧情线索库 (threads.yaml)
管理伏笔(planted)和收束(resolved)，防烂尾。字段：`id, description, planted_chapter/context, resolved_chapter/how, importance(major|minor|easter_egg), related_characters`。校验：major线索必须有推进、已resolved不重用、长期停滞触发遗忘警告。

### 模块四：时间线 (timeline.yaml)
按章节记录事件，防时间矛盾。字段：`chapter, events(timestamp, description, characters_involved, note)`。校验：角色不瞬移、时间流逝合理、回忆须标注。

### 模块五：世界规则 (rules.yaml)
**绝对不可违反的设定**。字段：`id, description, severity(hard|soft)`。hard 规则不可违反，soft 违反需合理理由。

### 模块六：剧情发散引擎
基于 14 种发散模式生成多个剧情方向。**核心原则：文似看山不喜平。**
- 14 种模式：D1线索咬合 | D2关系裂变 | D3禁区探索 | D4反转颠覆 | D5规则突变 | D6双线映照 | 🔑D7认知差悬念 | D8伪胜/败 | D9两难抉择 | 🔑D10倒计时压迫 | D11逆风孤战 | D12暗流涌动 | 🔑D13情感债回收 | D14渔线沉底
- 节奏模型：四层波浪+钩子链
- 详见 `references/divergence-patterns.md`、`references/auto-write-workflow.md`

#### 6.1 发散去重（防雷同）

> **问题**：AI 反复用同 2-3 种模式 → 章节雷同 → 读者疲劳。

**规则**：
1. **窗口检查**：生成方向前扫描最近 5 章的 `divergence_pattern`，最近 3 章用过的模式**禁止**作为主模式
2. **多样性下限**：10 章窗口内须用 ≥6 种模式，低于 = 强制注入未使用模式
3. **组合新鲜度**：相同模式组合 8 章内不可重复
4. **评分惩罚**：模式在最近 5 章每出现一次，新奇度 -1
5. **Stale 检测**：连续 3 章共享主模式 → 下一章必须用未使用池

**未使用池注入**：10 章内 >4 种模式未使用 → 下次必须包含 ≥1 个未使用方向

#### 6.2 世界观格局升级引擎

> **核心问题**：主角变强了，但冲突的质地没变——数字在涨，读者已经不在乎了。

**格局的本质不是「地图有多大」，而是「主角在乎的事有多深」。**

### 6.2.1 格局升级的四个维度

格局可以在以下任一维度上扩展，不必四个维度同时升级：

| 维度 | 低格局 | → 高格局 | 升级信号 |
|------|--------|----------|---------|
| **权力** | 个人武力/技能 | → 组织/制度/规则层面的力量 | 主角不再靠拳头，而是靠「规则本身」取胜 |
| **认知** | 眼前的真相 | → 世界运作的底层逻辑 | 主角发现「我一直以为的规则，只是别人想让我看到的」 |
| **关系** | 一对一的情感纽带 | → 群体/代际/命运层面的羁绊 | 主角的选择不再只影响自己，而是影响一群人、一个时代 |
| **代价** | 个人得失 | → 文明存亡/意义本身 | 输赢不再是「我能不能活」，而是「什么值得活」 |

**四个维度的关系**：不是阶梯，是光谱。一部小说可以主要在权力维度升级（如纯战斗流），也可以在关系和代价维度升级（如情感流）。好小说通常在 2-3 个维度上交替推进。

### 6.2.2 不同流派的格局路径

| 流派 | 典型升级轴线 | 示例 |
|------|-------------|------|
| 修仙/玄幻 | 权力：外门→内门→掌门→飞升 / 认知：功法→天道→宇宙真相 | 《凡人修仙传》从七玄门到灵界 |
| 都市/商战 | 权力：个人能力→团队→公司→行业规则 / 关系：对手→盟友→利益共同体 | 从摆摊到改写行业标准 |
| 悬疑/推理 | 认知：个案真相→连环模式→体制黑幕→人性本质 | 从破一个案子到发现「所有案子都是同一个源头」 |
| 言情/情感 | 关系：好感→羁绊→命运共同体 / 代价：个人幸福→对方的人生→两难抉择 | 从「我喜欢他」到「我们的选择决定了很多人的命运」 |
| 科幻 | 认知：技术原理→文明级应用→人类定义 | 从造一个AI到「什么是人」 |
| 末世/生存 | 权力：活下去→建立据点→重建文明 / 代价：个人→团队→人类火种 | 从活着到「为什么值得活」 |

### 6.2.3 格局升级铁律

1. **不可原地踏步**：同一维度的冲突质地不可停留超过 15 章。如果主角连续 15 章在「用同样方式解决同样级别的问题」→ 格局停滞。
2. **红色药丸时刻**：每次格局跨越必须有一个信息发现——主角（和读者）同时意识到「旧世界只是更大世界的子集」。不是旁白告知，是主角亲眼看到。
3. **能力不带格局 = 数字游戏**：新能力如果不解锁新的冲突类型，只是在旧战场上赢得更快——这是最隐蔽的注水。
4. **旧冲突不可回退**：格局升级后，旧层级的敌人不再是威胁。T1 的反派到 T3 必须变盟友/路人/笑料——反复纠缠说明故事在倒退。
5. **跨维度震荡**：在主要升级维度之外，偶尔在其他维度上打开一扇窗——给读者「还有更大的东西在后面」的预感。

### 6.2.4 升级后自检

```
□ 这次升级后，主角最在乎的东西变了吗？（不仅变强了，是变了）
□ 有没有一个场景，让旧版本的主角根本无法理解现在的冲突？
□ 是否存在新的冲突类型，在此之前完全不可能出现？
□ 之前的敌人还构成威胁吗？（构成 = 格局没升级）
```

**实操 Top 7+陷阱速记**：
0. **章节正文 write_file 大内容会超时**（≥800字）——用 `execute_code` 内 `from hermes_tools import write_file` 间接写入，不要直接调用 write_file 工具写章节正文。状态文件(yaml)不受影响
1. **arc-tracker 每章必更新**，否则 rhythm 退化为空
2. **状态文件不会自动更新**，每5章强制回写 characters/threads/debts
3. **consistency_checker 报全绿可能是空文件**——先确认数据有意义
4. **阶段名必须与 automation-config 匹配**，否则 rhythm 返回空
5. **跨会话必须先同步再写作**——详见 `references/session-resumption-playbook.md`
6. **commit_chapter 不自动切换弧线阶段**——跨越阶段时必须手动 patch arc-tracker.yaml 的 current_phase/phase_start_chapter/phase_chapters_elapsed
7. **consistency_checker 无 --deep 参数**——`check` 自动按章节号决定深浅，`audit` 做全局审查
8. **consistency_checker 兼容 dict/list 两种 YAML 格式**(v2.1.2+)——characters.yaml 和 relationships 字段无需特定格式
9. **章节文件名必须零填充**——`ch080.md` 不是 `ch80.md`，`write_file` 路径务必用 `ch{NNN}.md` 三位格式
10. **commit_chapter.py 自动发现章节文件**(v2.2+)--省略`--count-words`时自动按`--chapter`拼接`chapters/chNNN.md`路径
11. **consistency_checker 同规则折叠**(v2.2+)--同类🟢建议>5条自动折叠为一行汇总（`--verbose`查看全部）
12. **auto_sync.py 同步检测**(v2.2+)--`auto_sync.py --dir <项目>`一键检测characters/threads/arc-tracker/timeline/debts落后情况
13. **每章写完必须 git commit & push**(v2.4+)——commit 消息格式 `ch{NNN}: 标题`，卷完结加 `[卷完结]`，全书完结加 `[全书完结]`
14. **跨会话恢复先 git pull**(v2.4+)——其他会话可能已推送新章节，不 pull 就写会产生冲突
15. **每章前300字必须有冲突/行动**(v2.5+)——参见 §9.1 叙事克制，环境描写、背景回顾开篇都是违规
16. **大高潮超过5000字必须分拆**(v2.5+)——参见 §9.2，一口气写完高潮是叙事资产浪费
17. **前10章主角被动反应不超2章连续**(v2.5+)——参见 §9.3 能动性原则，2章连续被动必须下章补主动，3章连续一票否决
18. **commit_chapter.py --thread-advance 对不存在的线索ID静默跳过**——显示"⚠️ 线索 XXX 未找到，跳过"但不报错。提交前确认线索ID与threads.yaml一致；含"+"号的复合线索名会被跳过（parser局限）
19. **volume_transition.py 只做机械重置，不创建大纲/角色/线索**——后续8步手动清单见§7.5.1，遗漏任何一步都会导致新卷写作卡壳
20. **write_file 大内容超时**——章节正文超1200字时，直接调用 write_file 工具可能触发 stream timeout（~8K token 上限）。解决：用 `execute_code` 调用 `hermes_tools.write_file`，内容可分2-3段拼接写入。不要用 delegate_task（重试机制与小说写作不兼容）。示例：
    ```python
    from hermes_tools import write_file
    write_file(path="chapters/ch001.md", content=ch1_part1)
    # 或分两段：先写前半，再 read_file 取出 + 拼接后半 + write_file 覆写
    ```
21. **read_file 行号前缀污染**——`read_file()` 返回 `"N|内容"` 格式（带行号前缀），如果将此输出拼接到新文件内容中再 write_file，YAML/代码语法必毁。**绝对禁止 read→拼接→write 回环**。全量重写用 `terminal heredoc`（`cat > file.yaml`）或 `execute_code` 内直接 `write_file` 全量覆盖，不要读取旧内容拼接。
22. **YAML 状态文件用 terminal heredoc 写入**——characters.yaml/threads.yaml 等状态文件的全量重写，用 `terminal` 命令 `cat > path.yaml` 的 heredoc 方式最可靠（避免 Python 引号嵌套和 read_file 污染）。多角色时分2-3次 `cat >>` 追加。每次写完后 `python3 -c "import yaml; yaml.safe_load(open('file'))"` 验证。
23. **execute_code 写章节正文也有上限**——单次 execute_code 内的章节正文不宜超过~1200汉字，否则整个工具调用可能因参数超8K tokens而流式超时。章若需更多字数，分两次 execute_code 调用分别写前半和后半。
24. **卷间 characters.yaml 全量重写必须用 heredoc**——不要在 execute_code 内用 read_file 读取再拼接——行号前缀会永久污染文件，正则清理也不可靠（多行压缩/缩进丢失）。正确做法：`terminal` heredoc `cat > characters.yaml` 分批写入每个角色。

### CLI 工具速查（v2.1 新增）

| 命令 | 用途 | 示例 |
|------|------|------|
| `commit_chapter.py` | 一键章后状态更新（替代4-6次patch） | `--dir . --chapter 42 --words 3500 --pattern "D5+D9" --hook 焦虑型 --tension 8` |

### 7.10 有机冲刺模式（大纲驱动）

当卷大纲足够详细时，跳过 CLI 直接有机选择方向，速度是脚本模式的2-3倍。

**前提**：✅6大种子(主线×1+谜题×2+抉择×2+情感×1) ✅弧线阶段表 ✅设定速查表 ✅前卷状态已完成过渡

**零CLI循环（v2.2 commit_chapter 驱动）**：

> ⚠️ **核心原则：用户说"继续"=不间断冲刺。不要暂停问问题、不要输出统计表、不要确认。写就完了。**

```
1. 心算确认阶段 → 选择发散模式
2. write_file: 写章节正文 ch{NNN}.md（路径必须三位零填充！ch080.md ✅ ch80.md ❌）
3. commit_chapter.py --dir DIR --chapter N --title "标题" \
     --pattern "D5+D9" --hook 焦虑型 --tension 8 \
     [--thread-advance "T015:developed"]
   ↑ --count-words 已省略，v2.2+自动拼接 chapters/chNNN.md
4. 每3-5章: consistency_checker.py check DIR
5. 每5章: patch characters.yaml + emotional-debts.yaml
6. 每3-5章批量 git push: git add -A && git commit -m "ch{NNN}-{MMM}: 标题批" && git push origin main
   ↑ 批量提交比逐章push更高效。commit消息可用范围或最终章号标注
   ↑ 单章commit格式：ch{NNN}: 标题；批量：ch{NNN}-ch{MMM}: 批次提交
7. 继续下一章（不停顿、不问用户）
```
实测：90章连续写作，零手动patch，零一致性冲突

**冲刺模式字数管理**：连续冲刺10+章时字数会在1500-4500之间波动。稳定方法：
- 大纲每章标注预估字数（如"本章3000字+关键场景"）
- 每5章用 `auto_write.py wordcount --dir <目录>` 校准
- 字数缩水严重时在下一章方向中加入细节类模式（D6双线映照/D13情感回收）

**批量写作优化**（实测效率提升2-3倍）：连续冲刺时，可一次 invoke 2-3个 `write_file`（ch080/ch081/ch082），然后用一次 `commit_chapter.py` 逐章提交，最后批量 `git push`。减少上下文切换开销，但注意：
- 批量写作会降低每章的断章自检质量，高风险章节（高潮/反转/角色死亡）仍应单章写
- 批量 git push 的 commit 消息用范围格式：`ch080-ch082: 批次提交`

**与脚本模式切换**：卡住→回退CLI | 跨卷→必须CLI做过渡

**卷完结流程**：
1. `consistency_checker.py audit <目录>` 全局审查
2. `auto_write.py wordcount --dir <目录> --update` 精确字数
3. 检查 threads.yaml 全部 major 线索已 resolved
4. patch arc-tracker.yaml current_phase 为最终阶段
5. 输出完本统计表（章节/字数/线索收束/冲突数）

详细指南：`references/volume-outline-guide.md`

冲刺模式可通过 automation-config.yaml 的 sprint 部分启用。设置 sprint.enabled: true 跳过 rhythm/score，consistency 间隔自动放宽。

```

3. ☐ 全量重写 threads.yaml
   - 封卷旧线索：status=resolved, payoff_chapter=实际收束章节
   - 播种新线索：至少3条 active（1 main + 1 puzzle + 1 character_arc）

4. ☐ 全量重写 emotional-debts.yaml

   - 播种新情感债（每卷至少2-3条）

5. ☐ 更新 world.yaml
   - 新增本卷场景（每个至少: id/name/type/location_type/description/atmosphere/danger_level）
   - 新增世界规则（如有新设定的硬规则）

7. ☐ 一致性确认
   - 运行 consistency_checker.py check 确认无硬冲突

## 模块九：叙事纪律与读者情绪管理

### 9.1 叙事克制（全章通用）

- **300字入局**：每章前300字必须进入冲突/行动，禁止环境描写/背景回顾开篇。
- **背景融入**：禁止超过200字的纯设定段落。用"主角需要这个设定"的场景引入，而非"作者想告诉读者"。
- **目标驱动**：主角从第一章起必须有近期目标。3章内无主动行动 → 补上主动目标。

### 9.2 断章与高潮控制

- **在该断的地方断**：好章结尾是"事情正到最关键处"，不是"事情解决了"。断章黄金位置：决策前/真相前/冲突前/不安信息刚出现/角色踏入危险不知情。
- **大高潮分拆**：超过5000字→分拆为2-3章（高潮前夜→高潮爆发→高潮余波）。
- **断章自检**：读者感觉"好吧"非"然后呢？"→失败。好的断章让读者带着1-2个未解问题离开。

### 9.3 早期情绪保护（前10章）

**前10章不虐主**。红线：彻底击败无反击/剥夺核心能力/严重羞辱无反击/重要盟友死亡。连续2章被动→下章必须主动（第3章被动=一票否决）。

前10章主动行动占比≥60%。爽度密度加倍（每2章≥1中爽点）。第10章必做情绪审查。

> 完整规则：300字入局/背景融入判定标准/高潮分拆节奏表/10章情绪审查清单 → `references/narrative-discipline.md`

## 模块十：爽度引擎 + 画面感引擎

### 10.1 快节奏四原则

| 原则 | 规则 | 违规检测 |
|------|------|---------|
| **零废段** | 每段至少完成：推进剧情/制造冲突/揭示信息/创造爽点 | 删掉不影响主线=废段 |
| **场景快切** | 场景切换不超过3句过渡 | 超过3句=压缩 |
| **对话即角力** | 每句对话推进信息/施压/反击/暴露弱点。纯寒暄≤2句 | 超过3句纯寒暄=砍 |
| **行动代心理** | 用行动和对话展现内心。心理独白>150字且无冲突穿插=违规 | 拆散分布到行动中 |

### 10.2 爽度引擎

**六大爽型**（每5章≥3种）：装逼打脸/逆风翻盘/智商碾压/实力碾压/夺宝抢机缘/人心归附

**密度规则**：每3章≥1中爽点 | 每5章≥1大爽点 | 弧线高潮章≥1极爽点 | 连续2章零爽→违规告警（3章零爽=一票否决）

**爽点三步**：压制/困境（憋气）→ 转折/蓄力（合理翻盘，有伏笔支撑）→ 释放/碾压（快准狠，2段内完成）

**旁观者三层**（大爽点≥2层）：①直接对手崩溃/恐惧 ②同级旁观者震惊/重新评估 ③高手/前辈动容/忌惮

### 10.3 画面感引擎

**六大技法**：

| 技法 | 核心 |
|------|------|
| 镜头语言 | 远景/中景/特写/推/拉/跟拍，重要场景「远→中→特」切换 |
| 感官五重 | 重要场景≥3种感官，高潮5种全满。禁用「他很害怕」→ 用身体/感官说情绪 |
| 动作微分解 | 关键动作拆成1-3秒慢动作，适用一招制敌/关键反转/死亡瞬间 |
| 环境投射 | 用天气/光线/空间说情绪。压迫→低矮天花板，觉醒→光线涌入 |
| 物件象征 | 关键物件走完三阶段：首次出现（功能引入）→再次出现（叠加情感）→最终出现（封印使命） |
| 声画对位 | 声音和画面矛盾时张力最大。身后人声鼎沸，他只听见心跳 |

### 10.4 爽度+画面感联合规则

- **铁律一**：爽点必须有画面（≥2种技法呈现）。「他一拳打倒对手」=新闻，不是小说。
- **铁律二**：画面必须服务叙事。删掉后爽感不减的描写=废画面。
- **冲突优先级**：爽度 > 画面感 > 文笔修饰

### 10.5 每章自检

```
□ 【爽度密度】3章内≥1中爽？5章内≥1大爽？
□ 【三步构建】压制→转折→释放完整？
□ 【旁观者】大爽点≥2层旁观者反应？
□ 【类型轮换】近5章类型有变化？（不可连3次同类型）
□ 【镜头】「远→中→特」或「特→中→远」镜头流程？
□ 【感官】重要场景≥3种感官？高潮5种？
□ 【环境】环境描写服务情绪？删掉后受损？
□ 【物件】关键物件重复出现并叠加含义？
```

> 完整技法：六大爽型详细写法要点/反模式、六种镜头类型含示例、感官五重对比表、动作微分解示例、环境情绪投射表、物件象征三阶段写法、声画对位示例 → `references/satisfaction-cinema-details.md`

## 模块十一：散文质量层（P1·中文网文版）

> **核心前提**：这是网文，不是纯文学。散文质量服务节奏和爽度，而非文学野心。

### 11.1 五感平衡

| 感官 | 目标占比 | 说明 |
|------|---------|------|
| 视觉 | ~40% | 仍占主导，但不再垄断 |
| 听觉 | ~25% | 声音、寂静、对话节奏 |
| 触觉 | ~20% | 质感、温度、身体感受 |
| 嗅觉 | ~10% | 最强情绪触发感官——用来强化情感记忆 |
| 味觉 | ~5% | 在高潮/亲密场景尤其有效 |

**规则**：每章≥3种感官，新场景入场头500字5种全触发，打斗场景≥4种（视觉+听觉+触觉+嗅觉必含）。

### 11.2 句式变化

- **长度变化**：3句+长句（>15字）后插1短句（<8字）。短句制造冲击。
- **开头变化**：禁止3句+连续以相同结构开头（主-谓、人称代词、「XX是XX」）。
- **段落节奏**：禁止4段+连续纯叙述无对话/动作。
- **战斗句式**：打斗场景句子≤15字——短句=速度=冲击。

### 11.3 Show vs Tell

**铁律**：POV角色主要情绪≥60%用身体/动作/环境展示，不直接命名。

| ❌ 讲述 | ✅ 展示 |
|--------|--------|
| 她很愤怒。 | 她的指节在茶杯上攥得发白。瓷器发出细碎的悲鸣。 |
| 他感到恐惧。 | 他吐出的那口气变成了嘶哑的口哨。背已经贴到了墙上。 |

**红旗**：「感到XX」「很XX」「觉得XX」连续出现3次 → 强制修订。

### 11.4 对话工艺

- **交锋式**：每句对话推信息/施压/反击/暴露弱点。纯寒暄≤2句。
- **动作节奏**：≥50%的对话标签用动作（而非「XX说」）。
- **人物声纹**：每个角色有辨识度——词汇量/句长/口头禅/回避话题。
- **禁止解说式对话**：「如你所知……」型。

### 11.5 修辞新鲜度

- **每2000字≥1个新鲜比喻**，每章最多3个。
- **禁止陈词**：禁用「快如闪电」「冷若冰霜」「心如刀割」等所有成语级比喻。
- **比喻域统一**：同一段内不混用多域比喻。

> 完整规则含示例/反模式/自检清单 → `references/prose-quality-guide.md`

## 模块十二：章节解剖引擎（P0·每章必检）

> **核心问题**：AI能把情节节点总结成500字，但不知道怎么扩展成完整章节。

### 12.1 章节八元素（每章≥5/8）

| # | 元素 | ~字数 | 作用 |
|---|------|-------|------|
| A | **入局钩子** | 100-200 | 半途入局——动作开篇/感官冲击/对话直入。无铺垫。 |
| B | **行动序列** | 400-600 | 核心事件。战斗/谈判/发现/对峙。Show，不Tell。 |
| C | **反应段落** | 200-300 | 主角消化刚发生的事。身体/情绪/策略三重反应。 |
| D | **后果涟漪** | 200-300 | 行动如何改变局面？他人如何反应？新问题从旧答案中诞生。 |
| E | **配角时刻** | 200-300 | 非POV角色有意义的行动/发言。展示世界不围着主角转。 |
| F | **世界质感** | 100-200 | 感官细节/环境描写——**融入行动，非独立段落**。 |
| G | **子线推进** | 200-300 | 至少1条副线向前移动一步。 |
| H | **断章钩子** | 100-200 | 悬念/反转/追问/情感重击。**迫使读者点开下一章**。 |

### 12.2 六大扩容技法

当章节太薄时按顺序注入：**反应场景** → **旁观者视角** → **感官铺层** → **微型闪回** → **势力涟漪** → **代价对话**

### 12.3 章节密度规则

- 2+场景 / 2+角色有意义互动 / 1+感官变化 / 1+新信息 / 1+决策做出
- 缺失3/5要素 → 章节太薄，无论字数多少。

> 完整扩容技法含示例 → `references/chapter-anatomy-engine.md`

## 模块十七：角色深度引擎（P0）

### 17.1 信念系统（characters.yaml新增字段）

```yaml
belief_system:
  core_belief: "我相信什么（可能是错的）"
  deepest_fear: "我最怕什么"
  deepest_desire: "我最渴望什么"
  moral_line: "我绝不会做的事"
  self_deception: "我对自己说的谎"
```

**每卷至少1次信念挑战**——让角色面对与核心信念相悖的局面。

### 17.2 声音指纹

```yaml
voice_fingerprint:
  vocab_level: simple|normal|literary|technical
  sentence_length: short|medium|long|mixed
  signature_phrases: ["口头禅1", "口头禅2"]
  topics_gravitate: ["喜欢聊的"]
  topics_avoid: ["回避的"]
  speech_quirk: "独特说话方式"
```

**自检**：遮住角色名看对话——能分辨是谁在说话吗？

### 17.3 关系动态图

追踪关系的**张力和变化方向**（非简单「谁和谁是朋友」）：

```yaml
relationship_dynamics:
  - pair: ["主角", "导师"]
    surface: "尊敬/服从"
    subtext: "主角怀疑导师隐瞒了什么"
    trajectory: cooling  # warming/cooling/stable/volatile
```

### 17.4 角色弧线阶段

每个角色追踪阶段：`setup → build → crisis → transformation → resolution`。

### 17.5 人物过往（"填过往"技法）

> **核心原则**：没有无缘无故的善恶选择——每个人物的言行举止，都源于他的过往经历。过往四大来源：童年印记 | 原生创伤 | 执念遗憾 | 成长转折点。

**释放策略**：**千万不要一次性全盘托出**。分3-5次逐步释放，每次释放解答一个疑点又引出新疑问——既解释人物动机，又留有悬念让读者继续探究。释放时机：

| 时机 | 示例 |
|------|------|
| 面临相似情境 | 被背叛时→触发被背叛记忆 |
| 被迫做出选择 | 选择本身暴露过去 |
| 他人提及相关话题 | 反应异常引起注意 |
| 关键时刻闪回 | 战斗/决断前0.5秒回忆 |

**避坑**：回忆必须推动剧情或塑造人物——无意义回忆一律删除。每章最多1次回忆闪回。

### 17.6 人物细节（"加细节"技法）

> **核心原则**：动作的微表情、神态的细微变化、内心的挣扎纠结，远比直白的"他很生气""她很伤心"更有力量。这是画面感在角色层面的落地。

**三层次细节**：

| 层次 | 内容 | 密度规则 |
|------|------|---------|
| 动作微表情 | 手指攥紧/松开、瞳孔变化、喉结滚动 | 关键对话≥1层 |
| 神态变化 | 眼底温度升降、嘴角弧度、呼吸节奏 | 情感转折≥2层 |
| 内心碎片 | 1-2句碎片化心理活动（非长篇独白） | 高潮场面3层全开 |

**经典示例**：❌ "他不会骗我吧？"→ ✅ "他目光掠过她袖口里因紧张而攥紧的手指，眼底温度降了几分，语气却一如寻常：'你不会骗我的，对吗？'"

**避坑**：不必每句话都加动作——对抗性场面细节加倍，信息交换场面细节减半。只加对剧情和人物有意义的细节。

### 17.7 成长逻辑（"添成长"技法）

> **核心原则**：鲜活的人物从来不是一成不变的。无论主角还是配角，都会随着剧情发展而成长。

**成长四步链**：
```
困境/冲突 → 挣扎/反思 → 改变/突破 → 沉淀/蜕变
```

**反模式**：「无理由成长」（没有触发事件就变了）、「一蹴而就成长」（一章内完成蜕变）。

**三类角色成长线**：

| 类型 | 核心 | 示例轨迹 |
|------|------|---------|
| **主角成长（核心线）** | 贯穿主线的性格蜕变 | 自私→担当 / 懦弱→勇敢 / 迷茫→坚定 / 孤傲→有情 / 摆烂→开卷 |
| **配角成长（辅助线）** | 衬托主角+丰富故事层次 | 冲动徒弟→可靠助力 / 任性妹妹→温暖家人 |
| **反派成长（反向线）** | 不是"洗白"，是让恶有逻辑有根源 | 偏执→醒悟 / 被操控→找回自我 / 复仇→为救赎牺牲 |

**成长梯度铁律**：
- 懦弱的人不会突然无所畏惧——只会慢慢变勇敢
- 不是所有人都需要"完美成长"——有遗憾有不足更贴近真实
- 每次成长必须有对应困境/冲突作为触发事件
- 成长后允许偶尔反复（主角成长后依然会偶尔迷茫，只是学会了快速调整）

### 17.8 配角灵魂（"赋灵魂"技法）

> **核心原则**：配角不用是"工具人"。配角出彩的核心是"有辨识度、有记忆点、有价值"——不抢主角风头，又能让读者为了配角追更。

**四步打造有灵魂的配角**：

| 步骤 | 要点 | 说明 |
|------|------|------|
| **① 立人设** | 一个核心标签 + 1-2辅助细节 | 拒绝标签堆砌。标签可以是性格特质、行为习惯、口头禅、反差感 |
| **② 赋欲望** | 有"独立于主角"的欲望 | 哪怕很小很卑微。他的目标不是帮主角/害主角，而是自己的喜怒哀乐 |
| **③ 联主线** | 支线交织主线或暂时独立 | 感情流作品→发展副CP；剧情流作品→给配角写简单支线 |
| **④ 留白** | 不过度交代，让读者猜测回味 | 不必把过往/欲望/结局都写清楚 |

**配角生命周期**：作用结束 → 体面退场或加入新配角。不要少数几个配角反复横跳——那说明故事格局没升级。

**避坑三不**：不抢主角风头（大场面以主角为中心）| 不强行"洗白"或"黑化"（改变要符合逻辑）| 不重复冗余（不写人设雷同的配角）

## 模块十八：情感波动曲线引擎（P1）

> 一直爽=疲劳。一直虐=弃书。像音乐和弦进行一样设计：紧张→释放→紧张→释放，但每次天花板抬高。

| 分值 | 情感状态 | 典型事件 |
|------|---------|---------|
| +8~+10 | 极致快感 | 大爽点释放/宿敌覆灭 |
| +3~+7 | 正向推进 | 小胜利/新能力/盟友加入 |
| -3~+3 | 中性过渡 | 日常/铺垫/世界探索 |
| -7~-3 | 压力积累 | 被压制/失败/失去 |
| -10~-8 | 谷底绝望 | 重大损失/信念崩塌 |

**波形规则**：不可连续3章同象限 | 每次谷底反弹高于之前峰 | 前10章振幅小(-5~+7)，后期放开(-10~+10) | 卷高潮必须突破此前情绪天花板。

## 模块十九：对话潜台词引擎（P1）

### 19.1 三层信息不对称

| 层 | 内容 | 谁掌握 |
|----|------|--------|
| 表面层 | 嘴上说的字面意思 | 所有在场角色 |
| 意图层 | 说话者真正想表达/达成 | 说话者本人 |
| 潜藏层 | 说话者自己都没意识到的情感 | 只有读者（可能）知道 |

- 谁说真话？谁说假话？谁半真半假？
- 谁多知道什么？谁在试探？谁在伪装？
- 有没有一句话——不同角色听出不同意思？

**硬规则**：关键对话禁止「全员坦诚相待」——至少1个角色有隐藏动机。

## 快速开始

### 全自动模式（推荐）
1. **初始化**：`python3 scripts/auto_write.py init --name "书名" --mode auto`
2. **创建 GitHub 仓库**（注意顺序）：
   ```bash
   cd ~/novels/书名
   git init
   git config user.email "xxx@xxx.com" && git config user.name "name"
   git add -A && git commit -m "init: 项目初始化"
   git branch -m main
   gh repo create andy-develop/书名 --private --source=. --push
   ```
   如果 `gh repo create` 失败，可手动：创建远程仓库 → `git remote set-url origin https://USER:***@github.com/USER/书名.git` → `git push -u origin main`
3. **提供开头**：告诉 AI "全自动写《书名》，开头如下：[开头内容]"
4. **AI 自动完成**：发散→选择→写作→校验→Git同步，循环直至完成
5. **每章写完**：`commit_chapter.py` 一键更新状态 + `git add -A && git commit -m "ch{NNN}: 标题" && git push origin main`
6. **中途干预**：随时可以说"暂停"/"这一章我来选"/"节奏太慢"
7. **卷间过渡**：`volume_transition.py` 重置状态 + AI 手动完成大纲/角色/线索重写 + git push
8. **字数校准**：`auto_write.py wordcount --dir . --update`（CJK精确统计）
9. **完成**：AI 输出全书摘要、线索收束报告、润色建议 + 最终 git push `[全书完结]`

### 人机协作模式
1. **新建项目**：`python3 scripts/novel_state.py init --name "书名" --dir ~/novels/书名`
2. **创建 GitHub 仓库**：`cd ~/novels/书名 && git init && gh repo create andy-develop/书名 --private --source=. --push`
3. **填写设定**：编辑 state/ 下的 yaml 文件
