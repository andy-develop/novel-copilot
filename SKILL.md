# 长篇小说写作助手 (Novel Copilot)

超长篇小说（30万字+）的核心难题：**写到后面忘了前面**。本 skill 通过结构化状态追踪 + 逻辑校验 + 剧情发散 + **全自动写作引擎** + **叙事纪律引擎**，确保长篇不崩盘、不注水、不让读者弃书。

## 核心理念

> 写作是发散的，校验是收敛的。先发散，再校验，最后落笔。

**叙事三优先级**（冲突时高优先级不可妥协）：
- **P0 = 节奏 + 爽度 + 画面感** — 读者留不留得住、爽不爽、记不记得住，全靠这三条
- **P1 = 一致性 + 散文质量** — 不崩盘、不啰嗦
- **P2 = 大纲 + 字数 + 状态文件** — 工具而非枷锁

> 当 P0 与 P2 冲突时（如大纲要求铺垫但铺垫会拖慢节奏），**节奏爽感优先**，调整大纲而非牺牲节奏。

## 项目结构

> **每个小说项目 = 一个 GitHub 仓库**（默认 `andy-develop/书名`，private）。初始化时自动创建，每章写完自动 commit & push。

```
my-novel/                        # ← GitHub 仓库 andy-develop/my-novel
├── .git/                        # Git 版本控制
├── state/
│   ├── characters.yaml          # 角色状态库
│   ├── world.yaml               # 世界设定库
│   ├── threads.yaml             # 剧情线索库
│   ├── timeline.yaml            # 时间线
│   ├── rules.yaml               # 世界规则（不可违反的设定）
│   ├── arc-tracker.yaml         # 弧线追踪
│   ├── emotional-debts.yaml     # 情感债追踪
│   └── consistency-log.yaml     # 一致性校验日志
├── chapters/                    # 章节正文 ch{NNN}.md
├── outlines/                    # 弧线大纲 + 章节规划
├── automation-config.yaml       # 自动写作配置
└── novel-project.yaml           # 项目元信息

## 九大模块

### 模块一：角色状态库 (characters.yaml)
追踪角色当前状态，每次写到时自动校验。字段：`id, name, aliases, status(alive|dead|missing|sealed), death_chapter, death_cause, location, abilities, relationships, inventory, notes, last_appeared`。校验：死者无行动、物品唯一持有、能力增减有事件支撑、位置移动合理。

### 模块二：世界设定库 (world.yaml)
追踪地理、势力、能力体系。字段：`geography(id,name,type,location,status,destroyed_chapter), factions, power_system(name,ranks,rules)`。校验：已毁地点不再引用、势力变动有事件、能力使用不违反规则。

### 模块三：剧情线索库 (threads.yaml)
管理伏笔(planted)和收束(resolved)，防烂尾。字段：`id, description, planted_chapter/context, resolved_chapter/how, importance(major|minor|easter_egg), related_characters`。校验：major线索必须有推进、已resolved不重用、长期停滞触发遗忘警告。

### 模块四：时间线 (timeline.yaml)
按章节记录事件，防时间矛盾。字段：`chapter, events(timestamp, description, characters_involved, note)`。校验：角色不瞬移、时间流逝合理、回忆须标注。

### 模块五：世界规则 (rules.yaml)
**绝对不可违反的设定**。字段：`id, description, severity(hard|soft)`。hard 规则不可违反，soft 违反需合理理由。

### 模块六：剧情发散引擎
基于 14 种发散模式生成多个剧情方向。**核心原则：文似看山不喜平。**
- 14 种模式：D1线索咬合 | D2关系裂变 | D3禁区探索 | D4反转颠覆 | D5规则突变 | D6双线映照 | 🔑D7认知差悬念 | D8伪胜/败 | D9两难抉择 | 🔑D10倒计时压迫 | D11逆风孤战 | D12暗流涌动 | 🔑D13情感债回收 | D14渔线沉底
- 节奏模型：四层波浪+钩子链
- 详见 `references/divergence-patterns.md`、`references/auto-write-workflow.md`

## 模块七：自动写作引擎

> 执行协议：`references/auto-writing-execution-protocol.md` | 会话恢复：`references/session-resumption-playbook.md` | 有机冲刺：§7.10 + `references/volume-outline-guide.md`

```
开头/种子 → 初始化项目

- **7.3 循环详解**：读取→发散→评分→写作→更新→门控→Git同步→终止，8步完整流程 → `references/scoring-and-selection.md`
- **7.4 节奏自调节**：张力检测(连续同钩/高张疲劳/低张拖沓) + 弧线自动推进 → `references/scoring-and-selection.md`
- **7.5 长线管理**：跨弧线衔接 + 卷间过渡 + 终局管理 → `references/scoring-and-selection.md`
- **7.x 设定体系**：体系化设定速查 → `references/setting-system-tracking.md`
- **7.7 实操陷阱**：24条验证陷阱（含v2.1 commit_chapter工作流、字数统计、YAML引号嵌套等）→ `references/auto-write-practical-tips.md`
- **7.8 状态同步**：会话恢复必做 + 过期检测信号 + 高频/批量更新策略 → `references/auto-write-practical-tips.md`
- **7.9 方向融合**：TOP1+TOP2融合策略 + 实战案例 + 节奏参数 → `references/scoring-and-selection.md`

**实操 Top 7 陷阱速记**：
1. **arc-tracker 每章必更新**，否则 rhythm 退化为空
2. **状态文件不会自动更新**，每5章强制回写 characters/threads/debts
3. **consistency_checker 报全绿可能是空文件**——先确认数据有意义
4. **阶段名必须与 automation-config 匹配**，否则 rhythm 返回空
5. **跨会话必须先同步再写作**——详见 `references/session-resumption-playbook.md`
6. **commit_chapter 不自动切换弧线阶段**——跨越阶段时必须手动 patch arc-tracker.yaml 的 current_phase/phase_start_chapter/phase_chapters_elapsed
7. **consistency_checker 无 --deep 参数**——`check` 自动按章节号决定深浅，`audit` 做全局审查
8. **consistency_checker 兼容 dict/list 两种 YAML 格式**(v2.1.2+)——characters.yaml 和 relationships 字段无需特定格式
9. **章节文件名必须零填充**——`ch080.md` 不是 `ch80.md`，`write_file` 路径务必用 `ch{NNN}.md` 三位格式
10. **commit_chapter.py 自动发现章节文件**(v2.2+)--省略`--count-words`时自动按`--chapter`拼接`chapters/chNNN.md`路径
11. **consistency_checker 同规则折叠**(v2.2+)--同类🟢建议>5条自动折叠为一行汇总（`--verbose`查看全部）
12. **auto_sync.py 同步检测**(v2.2+)--`auto_sync.py --dir <项目>`一键检测characters/threads/arc-tracker/timeline/debts落后情况
13. **每章写完必须 git commit & push**(v2.4+)——commit 消息格式 `ch{NNN}: 标题`，卷完结加 `[卷完结]`，全书完结加 `[全书完结]`
14. **跨会话恢复先 git pull**(v2.4+)——其他会话可能已推送新章节，不 pull 就写会产生冲突
15. **每章前300字必须有冲突/行动**(v2.5+)——参见 §9.1 叙事克制，环境描写、背景回顾开篇都是违规
16. **大高潮超过5000字必须分拆**(v2.5+)——参见 §9.2，一口气写完高潮是叙事资产浪费
17. **前10章主角被动反应不超2章连续**(v2.5+)——参见 §9.3 能动性原则，2章连续被动必须下章补主动，3章连续一票否决
18. **commit_chapter.py --thread-advance 对不存在的线索ID静默跳过**——显示"⚠️ 线索 XXX 未找到，跳过"但不报错。提交前确认线索ID与threads.yaml一致；含"+"号的复合线索名会被跳过（parser局限）
19. **volume_transition.py 只做机械重置，不创建大纲/角色/线索**——后续8步手动清单见§7.5.1，遗漏任何一步都会导致新卷写作卡壳

### CLI 工具速查（v2.1 新增）

| 命令 | 用途 | 示例 |
|------|------|------|
| `commit_chapter.py` | 一键章后状态更新（替代4-6次patch） | `--dir . --chapter 42 --words 3500 --pattern "D5+D9" --hook 焦虑型 --tension 8` |
| `volume_transition.py` | 卷间机械状态重置+AI待办清单 | `--dir . --volume 5 --title "根" --first-chapter 71` |
| `auto_write.py wordcount` | 扫描全部章节统计中文字数 | `--dir . [--update]`（无--update仅打印） |
| `auto_write.py score --mode 3dim` | 3维简化评分(节奏40%/线索35%/安全25%) | `--d1 "D7,4,5,4"` |
| `consistency_checker.py --verbose` | 显示全部建议含已抑制的 | 默认抑制连续3次🟢建议+同规则>5条折叠 |
| `auto_sync.py --dir .` | 检测状态文件同步差距 | 输出characters/threads/arc-tracker/timeline落后章数 |

`commit_chapter.py` 高级用法：
- `--count-words chapters/ch042.md` 自动统计中文字数，覆盖 `--words`（v2.2+：省略此参数时自动按`--chapter`拼接路径）
- `--thread-advance "T015:developed,T018:resolved:42"` 批量推进/收束线索
- `--dry-run` 预览变更不写入

### 7.10 有机冲刺模式（大纲驱动）

当卷大纲足够详细时，跳过 CLI 直接有机选择方向，速度是脚本模式的2-3倍。

**前提**：✅6大种子(主线×1+谜题×2+抉择×2+情感×1) ✅弧线阶段表 ✅设定速查表 ✅前卷状态已完成过渡

**零CLI循环（v2.2 commit_chapter 驱动）**：

> ⚠️ **核心原则：用户说"继续"=不间断冲刺。不要暂停问问题、不要输出统计表、不要确认。写就完了。**

```
1. 心算确认阶段 → 选择发散模式
2. write_file: 写章节正文 ch{NNN}.md（路径必须三位零填充！ch080.md ✅ ch80.md ❌）
3. commit_chapter.py --dir DIR --chapter N --title "标题" \
     --pattern "D5+D9" --hook 焦虑型 --tension 8 \
     [--thread-advance "T015:developed"]
   ↑ --count-words 已省略，v2.2+自动拼接 chapters/chNNN.md
4. 每3-5章: consistency_checker.py check DIR
5. 每5章: patch characters.yaml + emotional-debts.yaml
6. 每3-5章批量 git push: git add -A && git commit -m "ch{NNN}-{MMM}: 标题批" && git push origin main
   ↑ 批量提交比逐章push更高效。commit消息可用范围或最终章号标注
   ↑ 单章commit格式：ch{NNN}: 标题；批量：ch{NNN}-ch{MMM}: 批次提交
7. 继续下一章（不停顿、不问用户）
```
实测：90章连续写作，零手动patch，零一致性冲突

**冲刺模式字数管理**：连续冲刺10+章时字数会在1500-4500之间波动。稳定方法：
- 大纲每章标注预估字数（如"本章3000字+关键场景"）
- 每5章用 `auto_write.py wordcount --dir <目录>` 校准
- 字数缩水严重时在下一章方向中加入细节类模式（D6双线映照/D13情感回收）

**批量写作优化**（实测效率提升2-3倍）：连续冲刺时，可一次 invoke 2-3个 `write_file`（ch080/ch081/ch082），然后用一次 `commit_chapter.py` 逐章提交，最后批量 `git push`。减少上下文切换开销，但注意：
- 批量写作会降低每章的断章自检质量，高风险章节（高潮/反转/角色死亡）仍应单章写
- 批量 git push 的 commit 消息用范围格式：`ch080-ch082: 批次提交`

**与脚本模式切换**：卡住→回退CLI | 跨卷→必须CLI做过渡

**卷完结流程**：
1. `consistency_checker.py audit <目录>` 全局审查
2. `auto_write.py wordcount --dir <目录> --update` 精确字数
3. 检查 threads.yaml 全部 major 线索已 resolved
4. patch arc-tracker.yaml current_phase 为最终阶段
5. 输出完本统计表（章节/字数/线索收束/冲突数）

详细指南：`references/volume-outline-guide.md`

冲刺模式可通过 automation-config.yaml 的 sprint 部分启用。设置 sprint.enabled: true 跳过 rhythm/score，consistency 间隔自动放宽。

### 7.5.1 卷间过渡完整清单

> `volume_transition.py` 只做机械重置（arc-tracker/consistency-log/novel-project），以下手动步骤 **必须全部完成** 才能开始新卷写作：

```
1. ☐ 创建卷大纲 outlines/volumeN-outline.md
   - 六大种子（主线×1 + 谜题×2 + 抉择×2 + 情感×1）
   - 弧线阶段规划表（阶段|章节范围|核心事件|张力|发散模式）
   - 前5-10章逐章规划（模式/核心/钩子/张力）

2. ☐ 全量重写 characters.yaml
   - 保留存活角色（更新 last_appeared/condition/abilities/relationships）
   - 新增本卷新角色（每个至少: id/name/status/abilities/power_level/relationships/key_traits）
   - 移除不再出场的角色或标记 status=absent
   - 每个 relationship 标注 since_chapter

3. ☐ 全量重写 threads.yaml
   - 封卷旧线索：status=resolved, payoff_chapter=实际收束章节
   - 播种新线索：至少3条 active（1 main + 1 puzzle + 1 character_arc）
   - 每条新线索必须有 planted_chapter/last_updated_chapter/key_points
   - 保留旧线索（status=resolved）供一致性校验回溯

4. ☐ 全量重写 emotional-debts.yaml
   - 结转未偿还的旧债（status=outstanding/partially_paid）
   - 播种新情感债（每卷至少2-3条）

5. ☐ 更新 world.yaml
   - 新增本卷场景（每个至少: id/name/type/location_type/description/atmosphere/danger_level）
   - 新增世界规则（如有新设定的硬规则）
   - 保留旧场景（可能跨卷复用）

6. ☐ 更新 timeline.yaml
   - 添加卷分界标记（chapter=上卷末/下卷首, event="卷N→卷N+1过渡"）
   - 清空本卷事件区（写作中更新）

7. ☐ 一致性确认
   - 运行 consistency_checker.py check 确认无硬冲突
   - 确认 arc-tracker.current_phase 与 automation-config 匹配

8. ☐ Git 提交
   - git add -A && git commit -m "state: 卷N→卷N+1过渡+状态文件全量更新 [卷N+1准备]"
   - git push origin main
```

**常见遗漏**：忘记更新 `since_chapter` 字段→关系时间线错乱；新线索缺 `key_points`→后续章节推进无锚点；world.yaml 缺 `danger_level`→禁区探索模式无参考基线。

## 使用流程

### 模式A：全自动写作
1. **会话恢复**：确认章节数与 novel-project.yaml 一致，不同步则先批量更新 → `references/session-resumption-playbook.md`
2. **初始化**：`auto_write.py init --name "书名" --dir ~/novels/my-novel --mode auto`
3. **提供开头**：AI 自动提取设定→填充状态→进入循环
4. **自动循环**：每章输出 📝完成/🎭模式/🎣钩子/📊进度/✅一致性
5. **章后提交**：`commit_chapter.py` 一键更新 novel-project/arc-tracker/threads/consistency-log（替代手动patch）
6. **中途干预**：随时"暂停"/"用D9"/"角色不该死"/"节奏太慢"/"跳到高潮"
7. **卷间过渡**：`volume_transition.py` 重置状态 + AI 手动完成大纲/角色/线索重写
8. **完成输出**：全书摘要 + 线索收束 + 一致性审计 + `wordcount --update` 精确字数

### 模式B：人机协作
1. **初始化**：`novel_state.py init --name "书名" --dir ~/novels/my-novel`
2. **写前校验**：`consistency_checker.py check ~/novels/my-novel`
3. **发散+评分**：AI 生成3-5方向含评分，你选方向，AI 写章节
4. **写后更新**：AI 自动提取事件→更新yaml→校验
5. **定期审查**：`consistency_checker.py audit ~/novels/my-novel`

## 模块八：开篇冲击力引擎（第1-2章专用）

> **铁律：读者给一本新小说的耐心只有两章。第一章抓不住，第二章留不住，就不会有第三章。**
> 第1-2章不是"铺垫"，是"战场"——必须在前3000-8000字内完成读者绑架。

### 8.1 第一章：致命开局

第一章必须同时完成 **三件事**，缺一不可：

| 必须完成 | 说明 | 常见错误 |
|----------|------|----------|
| **即刻冲突** | 开篇第一场景就有冲突/危机/异常，而非平静日常 | 用大段世界观介绍或角色独白开篇 |
| **角色锚定** | 读者在5分钟内知道主角是谁、想要什么、处境如何 | 主角模糊，读者到章末都不在乎这个人 |
| **悬念炸弹** | 章末抛出一个读者无法不追问的问题 | 温吞结尾，"明天又是新的一天"式收束 |

**第一章死亡模式**（出现任一即重写）：
- ❌ 大段设定介绍（世界观、历史、能力体系说明文）
- ❌ 角色醒来照镜子式开头
- ❌ "一切都很正常"式开篇（正常=无聊）
- ❌ 无冲突的日常描写超过500字
- ❌ 章末无紧迫悬念（"明天再说"式收束）

**第一章推荐发散模式（优先级排序）**：
1. **D7 认知差** — 读者立刻知道角色不知道的危险，30秒内制造焦虑
2. **D10 倒计时** — 开篇就倒计时，读者跟着紧迫
3. **D3 禁区** — 开场就在危险之地或正在做危险之事
4. **D12 暗流** — 看似正常的场景藏着3个微异常信号

### 8.2 第二章：升级绑架

第二章的任务：**把好奇变成放不下**。读者在第一章产生的兴趣，第二章必须转化为情感投资+认知焦虑。

| 必须完成 | 说明 |
|----------|------|
| **冲突升级** | 第一章的危机/冲突必须在本章加深或扩大，不可原位踏步 |
| **情感投资** | 让读者在乎——角色的困境必须有情感重量（不是信息重量） |
| **认知翻转** | 至少一个"原来看错了"的认知刷新，颠覆第一章建立的假设 |
| **次级悬念** | 至少抛出1条新的次级线索/问题，与主线悬念交叉 |

**第二章死亡模式**：
- ❌ 退回解释第一章发生了什么（读者已经知道了）
- ❌ 大段世界观补充（觉得第一章没讲够，第二章来补课）
- ❌ 危机降温（第一章紧张，第二章突然安全了）
- ❌ 新角色大段独白/背景介绍
- ❌ 章末张力低于第一章

### 8.3 第1-2章特殊评分规则

当 `current_chapter ≤ 2` 时，评分权重 **大幅调整**：

| 维度 | 标准权重 | 第1-2章权重 | 理由 |
|------|---------|------------|------|
| 翻页驱动 | 20% | **35%** | 开篇首要目标：让读者翻页 |
| 钩子强度 | 5% | **25%** | 章末钩子是开篇的生死线 |
| 节奏适配 | 25% | 10% | 开篇不需要适配弧线节奏，需要制造节奏 |
| 情感利用 | 15% | 15% | 不变，情感投资同样关键 |
| 线索推进 | 15% | 5% | 开篇线索尚未建立，推进权重降低 |
| 一致性安全 | 10% | 5% | 开篇设定少，冲突风险低 |
| 新颖度 | 10% | 5% | 新颖度重要但不如钩子生死攸关 |

```
Score_ch1-2 = 0.10*rhythm_fit + 0.35*drive_intensity + 0.05*thread_advancement
            + 0.15*emotional_utilization + 0.05*consistency_score + 0.05*novelty
            + 0.25*hook_strength
```

**额外一票否决**（第1-2章专用）：
- 翻页驱动 ≤ 2 → 直接否决，无论总分多高
- 钩子强度 ≤ 2 → 直接否决，章末钩子是开篇的命脉
- 章节前500字无冲突/异常/紧张 → 直接否决
- 第1-2章零爽点（无打脸/碾压/翻盘等任何爽感场景）→ 直接否决，只有悬念没有爽=开篇温吞

### 8.4 第1-2章写后冲击力审查（必做）

> ⚠️ 写完第1章和第2章后，**必须执行冲击力审查**。这不是可选项。

**审查清单**（全部通过才能继续写第3章）：

```
□ 开篇第一句/第一段是否有冲突/异常/紧迫感？
□ 读者在前300字内能否知道"这人想干什么"？
□ 前500字是否有至少一个"不对劲"的信号？
□ 章末钩子是否足够强——读者是否会"必须翻下一章"？
□ 如果撕掉最后一段，前面的内容是否还能留住读者？（检验内容本身的力量，而非仅靠钩子）
□ 情感锚点——读完后读者是否"在乎"主角的命运？
□ 是否有超过200字的纯设定/世界观说明段落？（有=删或融入冲突场景）
□ 张力曲线是否从头到尾单调？（必须至少有一个波峰波谷）
□ 第二章结尾张力是否≥第一章结尾张力？
□ 是否有"认知翻转"——读者在第二章是否发现第一章的理解是错的或不完整的？
```

**审查不通过的处理**：
- 任何一项不通过 → 回到写作阶段修改，不可跳过
- 连续两次审查不通过 → 重新发散第一章方向，从根源调整

### 8.5 首章钩子模板（灵感参考）

| 类型 | 开篇模式 | 示例 |
|------|---------|------|
| **行动中切入** | 主角正在做危险/紧张的事 | "刀锋划过锁骨的时候，林昭才意识到这不是演习。" |
| **倒置悬念** | 先给结果再回溯 | "三天后，整座城市都会知道他的名字。但此刻他只是在雨中等一个外卖。" |
| **认知炸弹** | 一句话颠覆常识 | "银电纹路在他手臂上浮起的瞬间，所有医学教科书都得重写。" |
| **异常日常** | 平静中藏着一根刺 | "骑士站点的排行榜今天更新了，但所有人都只盯着最下面那个名字——那个昨天还活着的人。" |
| **对话开篇** | 一句信息量爆炸的对话 | ""你的预知能力是0.05秒？""0.1秒。"他撒了谎。" |

## 模块九：叙事纪律与读者情绪管理

> **核心理念**：好的故事不靠字数取胜，靠每一页都让读者放不下。叙事纪律不是束缚，是让每一字都有杀伤力的武器。
> Module 8 管第1-2章的开篇冲击力，Module 9 管从第1章到最后一章的叙事质量和读者情绪节奏。

### 9.1 叙事克制（全章通用）

> 适用于**每一章**，不是只有开篇。Module 8 的死亡模式只管1-2章，以下规则管全书。

**法则一：300字入局**
- 每章开篇300字内必须进入剧情——直接展现冲突、行动或紧张感
- 禁止以环境描写、背景回顾、情绪独白、时间流逝概述开篇
- 写后自检：删掉本章前300字，如果后面的内容不受影响 → 那些字是废话，换掉
- 例：❌ "夜色渐深，城市的灯火如星河般铺展开来……" → ✅ "门被踹开的时候，林昭手里的杯子还没放下。"

**法则二：背景融入，不独白式罗列**
- 设定和世界观信息只通过角色的行动和冲突来展现，不单独罗列
- 禁止超过200字的纯设定/背景说明段落（任何章节，不只是开篇）
- **"纯设定段落"判定标准** — 一段文字同时满足以下3个条件即计入：
  1. 没有角色在行动（没人在做、在决定、在遭遇）
  2. 没有冲突或紧张感（无威胁、困境、悬念、情绪压力）
  3. 信息流向是"作者→读者"（非角色对话中获知、非角色困境中领悟）
- **字数计算**：从满足上述3条件的第一句开始，到任一条件被打破为止，累计中文字符数（含标点）
- **不计时的情况**：角色对话中的设定讨论 / 角色主动查阅时伴随行动的设定 / 设定信息后立即被角色行动或悬念打破（≤2句过渡不算打破）
- 技巧：用"主角需要用到某个设定"的场景引入设定，而非"作者想告诉读者"的场景
- 日常戏只在推进主线或深化关系时允许存在——与主线无关的日常就是水字数

**法则三：目标驱动，不慢热**
- 主角从第一章起必须有明确的**近期目标**（不是远大理想）
  - 近期目标 = "今天必须拿到那把钥匙"/"这周内找到失踪的妹妹"
  - 远期目标 ≠ 近期目标："成为最强的剑士"是远期，"今天必须通过入门试炼"是近期
- 每个弧线阶段主角都应有当前目标，目标完成后立即出现新目标
- 如果主角3章内没有主动行动（只是在被动反应）→ 目标感缺失，必须在下章补上主动目标
- **慢热 = 读者流失**：没有目标感的等待是读者最容易弃书的时刻

**写后叙事自检**（每章完成后快速过一遍）：
```
□ 前300字是否直接进入了剧情？有没有无意义的铺垫？
□ 有无超过200字的纯设定/背景说明段落？（判定标准：同时满足"无角色行动+无冲突+作者→读者信息流"的连续文段，累计中文字符含标点）
□ 主角在本章是否有主动行动或明确目标？
□ 本章内容删掉后，主线是否受损？不受损=可删
```

### 9.2 断章与高潮控制

> 钩子（Module 6 / divergence-patterns 层级三）决定"章末留什么悬念"，断章纪律决定"在哪里断"。两者应同时考虑：
> - 先确定断章的黄金位置（结构节点）
> - 再确认该位置产生的钩子是否满足类型轮换要求
> - 如果断章自然产生的钩子与钩子类型轮换冲突（如连续2章同一类型），则微调断章位置或在断章点补充次级钩子

**纪律一：正常断章——在该断的地方断**
- 每章末尾在叙事的自然呼吸点断开，而非把一个完整的情绪弧写完
- 好章的结尾不是"事情解决了"，而是"事情正到最关键处"
- **断章的黄金位置**：
  - 决策即将做出之前
  - 真相即将揭示之前
  - 冲突即将爆发之前
  - 一个令人不安的新信息刚刚出现
  - 角色正要踏入危险而不知情
- **反模式**：把一个完整的小故事（起承转合全包含）写完再断章——这叫流水账，不叫章

**纪律二：大高潮分拆法则——最值钱的叙事资产必须分拆使用**
- 不要在一个章节内写完大高潮的完整过程。高潮是全书最值钱的叙事资产，一口气写完 = 浪费
- **高潮分拆节奏**：
  | 阶段 | 内容 | 断章点 |
  |------|------|--------|
  | 高潮前夜 | 铺设最后张力，角色做出最终选择 | ✂️ 在选择即将做出时断 |
  | 高潮爆发 | 冲突正面碰撞，但不写完结果 | ✂️ 在最紧要关头断 |
  | 高潮余波 | 结果揭示 + 情感冲击 + 新悬念 | 此时可收束 |
- 如果预估高潮内容超过5000字 → **必须分拆为2-3章**
- 为什么：一次性写完大高潮，读者的情绪瞬间释放后立即萎靡；分拆使用则让高潮的张力贯穿多章，维持叙事弹性

**纪律三：断章自检**
- 读者看到本章最后一行时的感觉是"好吧"而非"然后呢？" → 断章失败
- 本章的情绪曲线从起点到终点是完整闭环（紧张→高潮→解决→平静）→ 断章过于完整
- **好的断章让读者带着1-2个未解的问题离开，而非带着满足感离开**

### 9.3 早期情绪保护（前10章）

> 读者在前10章对主角的认同感决定了一本书能否被追读。"欲扬先抑"的"抑"如果过重，读者不是在期待"扬"，而是在找下一本书。

**核心原则：前10章不虐主**

具体红线（前10章内禁止）：
- ❌ 主角被彻底击败且无法反击（可以败，但必须保留主动性或还手机会）
- ❌ 主角被剥夺核心能力/资源且短期内无法恢复
- ❌ 主角被严重羞辱且没有反击机会
- ❌ 主角的重要盟友在前10章死亡（除非死亡本身成为主角的驱动目标，且主角立即由此获得新的行动方向）
- ⚠️ 主角连续2章只被动承受不主动出击 → 下一章**必须**安排主动行动事件（连续第3章被动将触发方向一票否决）

允许的情况：
- ✅ 主角遇到困难但展现了智慧和韧性
- ✅ 主角暂时处于劣势但主动寻找破局之道
- ✅ 主角被挫败但立刻获得新的线索/机会
- ✅ 小挫折（被拒绝、被误解）——但2章内必须有反转

**"能动性"原则**：
- 前10章的主角行为中，主动行动占比应 ≥ 60%
- 主动行动 = 主角主动发起的行动、主动做出的选择、主动承担的风险
- 被动反应 = 被打、被骗、被追、被动接受安排
- 如果主角连续2章只被动反应 → 下一章必须安排主动行动事件（连续第3章被动将触发方向一票否决）

**第3-10章特殊评分调整**：

> Module 8 管第1-2章，本规则管第3-10章。当 `3 ≤ current_chapter ≤ 10` 时，评分额外约束：

| 条件 | 效果 |
|------|------|
| 方向中主角被严重压制且无主动空间 | 情感利用维度上限降为2分 |
| 方向导致主角失去核心能力 | 一票否决 |
| 方向让主角展现主动性和智慧 | 翻页驱动维度 +1（最高5） |
| 方向让主角获得阶段性胜利（哪怕小的） | 情感利用维度 +0.5 |
| 方向含爽点（打脸/碾压/翻盘） | 翻页驱动 +1，情感利用 +0.5 |
| 连续2章方向零爽点 | 翻页驱动 -1 |
| 前10章爽度密度加倍：每2章≥1中爽点 | 不达标时情感利用维度上限降为3分 |

**第10章写后情绪审查**（写完第10章时必做）：
```
□ 主角在前10章中是否有至少3次主动出击？
□ 主角是否遭遇过"无还手之力"的严重打压？（有=不合格，需补章修正）
□ 读者读到第10章时，是否"在乎"主角？还是"同情"主角？（在乎✅ 同情❌）
□ 主角是否从被动变主动了？（前5章可被动，后5章应越来越主动）
□ 前10章的整体情绪走向是否"低开高走"？（应该如此）
```

## 模块十：爽度引擎 + 画面感引擎

> **核心命题**：网文读者买单的不是文笔，是"爽"和"画面"。爽度决定读者充不充钱，画面感决定读者记不记得住。两者缺一不可——有爽无画面是新闻联播，有画面无爽是文艺片。叙事快节奏是爽的前提：不浪费一个字，读者才不会走神。

### 10.1 叙事快节奏——爽的前提

> 快节奏不是"快"，是"不浪费"。每一段都有杀伤力，读者才不会走神。

**快节奏四原则**：

**原则一：零废段**
- 每个段落必须至少完成以下之一——推进剧情 / 制造冲突 / 揭示信息 / 创造爽点。四者皆不满足 = 废段，删。
- 自检法：删掉该段，后面内容是否受影响？不受影响 = 废段。

**原则二：场景切换快**
- 场景切换不超过3句过渡。过渡和叙事同时完成。
- ❌ 「三天后，他到了京城。」→ ✅ 「京城的城墙在晨光中浮出来时，马车已经跑了三天。」
- ❌ 「会议结束后，他回到办公室。」→ ✅ 「会议室的门还没关上，他已经拨出了电话。」

**原则三：对话即角力**
- 每句对话必须至少做到：推进信息 / 施压 / 反击 / 暴露弱点。纯寒暄对话最多2句。
- ❌ 「你好。」「你好。」「今天天气不错。」「是啊。」→ ✅ 「你来迟了。」「我来了，这就够了。」

**原则四：行动代心理**
- 直接用行动和对话展现内心，不用大段独白解释角色在想什么。
- ❌ 「他感到一阵愤怒涌上心头，心想这个人太过分了」→ ✅ 「杯子在他手里碎了」
- 心理独白超过150字且无冲突/悬念穿插 → 违规，拆散分布到行动中

**节奏违规检测**（每章写后自检）：
```
□ 有无连续3段以上既无冲突又不推进主线的内容？（有=删减至1段或融入冲突）
□ 场景切换过渡是否超过3句？（超过=压缩）
□ 对话中是否超过3句纯寒暄/纯叙述无角力？（超过=砍）
□ 心理独白是否超过150字且无冲突/悬念穿插？（超过=拆散分布到行动中）
□ 整章是否有超过10%的文字删掉后不影响主线？（有=注水，精简）
```

### 10.2 爽度引擎

> 爽 = 读者的情感回报。没有回报的付出是受罪，不是故事。爽度是网文的硬通货。

#### 10.2.1 爽的六大类型

**每5章至少覆盖3种，不可只依赖单一类型。**

| 类型 | 核心 | 写法要点 | 反模式 |
|------|------|----------|--------|
| **装逼打脸** | 低估→碾压→震惊 | 压制越狠，反弹越爽；围观者震惊是催化剂 | 主角自夸无实力支撑；打脸无铺垫 |
| **逆风翻盘** | 绝境→转折→反杀 | 绝境要真（读者也觉得完了），转折要巧（1-2个伏笔回收），反杀要快 | 绝境假（读者知道肯定会赢）；转折凭空天降 |
| **智商碾压** | 对手自以为聪明→主角看穿→降维打击 | 展现对手的算计过程（让读者先悬着），再揭示主角早有预判 | 主角全知无逻辑；对手蠢到不值得碾压 |
| **实力碾压** | 越级挑战→一招秒杀→旁观者炸锅 | 必须先铺垫对手的可怕（ta有多强），秒杀才有多震撼 | 对手是杂鱼→碾压无意义 |
| **夺宝抢机缘** | 众人争夺→主角截胡→宝物立竿见影 | 争夺要热闹（人多竞争感强），截胡要干脆，宝物效果立刻可见 | 宝物到手3章不用，爽感消散 |
| **人心归附** | 展现实力/人品→前对手/前辈心服→获得助力 | 归附要有过程（不是一句话就服了），被归附者的分量决定爽度 | 廉价归附（路人甲跪喊大佬） |

#### 10.2.2 爽度密度规则

| 周期 | 最低爽度要求 | 爽点级别 |
|------|------------|---------|
| 每3章 | ≥1个中爽点 | 小打脸 / 小翻盘 / 小碾压 / 对手吃瘪 |
| 每5章 | ≥1个大爽点 | 核心装逼 / 重要翻盘 / 实力碾压 / 夺大机缘 |
| 弧线高潮章 | ≥1个极爽点 | 全书级反转打脸 / 越两级碾压 / 智商降维打击 |
| 连续2章零爽点 | ⚠️ 节奏违规 | 下章必须补爽，第三章仍零爽则一票否决 |

**爽度递增原则**：同一弧线内，每个大爽点的冲击力应**超过前一个**。读者对爽感有耐受性——同样的打脸两次就变无聊，必须每次升级（对手更强/场景更大/围观者更震撼）。

#### 10.2.3 爽点构建三步法

每个爽点必须经历完整三步，缺一步则爽感打折：

```
Step 1 压制/困境——让读者憋着一口气
  ├── 关键：压制必须充分，但时间不超1章（小爽点不超半章）
  ├── 手法：对手嘲讽轻视 / 主角被困被压 / 局势看似无解 / 围观者不看好的议论
  └── 禁忌：假压制（读者一眼看穿主角在演戏）→ 无憋屈则无释放

Step 2 转折/蓄力——主角展示真正实力或揭露隐藏筹码
  ├── 关键：转折必须合理（有伏笔支撑或逻辑自洽），不能天降
  ├── 手法：底牌揭开 / 能力觉醒 / 盟友赶到 / 对手弱点暴露 / 伏笔回收
  └── 禁忌：deus ex machina——无铺垫的外力拯救 = 作弊，读者不爽反而骂

Step 3 释放/碾压——爽点爆发，读者的憋屈瞬间释放
  ├── 关键：释放要快准狠，不超过2段完成碾压
  ├── 手法：一击制胜 / 一句话封喉 / 围观者震惊反应 / 对手崩溃/不可置信
  └── 禁忌：释放拖沓——打脸超过3段还没打完 = 爽感消散
```

#### 10.2.4 爽度的催化剂——旁观者反应

> **爽点的50%力量来自围观者的震惊/恐惧/崇拜/无法理解。** 主角自己爽不算爽，有人看着才是真爽。

**三层旁观者反应**（大爽点至少覆盖2层）：

| 层次 | 视角角色 | 反应类型 | 效果 |
|------|---------|---------|------|
| 第一层 | 直接对手 | 崩溃 / 恐惧 / 不可置信 | 确认碾压的真实性 |
| 第二层 | 同级别旁观者 | 震惊 / 重新评估 / 后怕 | 放大碾压的分量 |
| 第三层 | 高手/前辈 | 动容 / 重新评估 / 隐约忌惮 | 暗示主角的上限远不止此 |

**写法示例**：
```
第一层：那人倒在地上，嘴里的血沫还在冒，眼睛瞪着——他到死大概都不相信。
第二层：全场寂静了三秒。然后有人低声说：「那个三阶的……一招？」
第三层：高台上的老人缓缓放下了茶杯，目光从懒散变作了认真。
```

#### 10.2.5 爽度评分融入方向评估

| 条件 | 效果 |
|------|------|
| 方向含有大爽点（装逼/翻盘/碾压） | 翻页驱动 +1，情感利用 +0.5 |
| 方向含爽点且爽点有充分铺垫（前1-2章已有压制） | 翻页驱动额外 +0.5 |
| 连续2章方向零爽点 | 翻页驱动 -1 |
| 3章累计爽点为零 | 一票否决 |
| 爽点类型与近5章完全重复（如连3次打脸） | 新颖度上限降为2 |

### 10.3 画面感引擎

> 画面感 = 读者闭眼能看见。没有画面的文字是报告，不是小说。

#### 10.3.1 技法一：镜头语言

把每个场景当作一个镜头来写，明确"摄影机"的位置和运动。

| 镜头类型 | 写法 | 示例 |
|---------|------|------|
| **远景** | 建立空间感，1-2句 | 「城墙上只剩一个人影，脚下是万丈深渊。」 |
| **中景** | 展现动作和互动，构建主要叙事 | 「他踩着碎石一步步往上走，每一步都带出血痕。」 |
| **特写** | 放大关键细节，制造冲击力 | 「那只手伸过来的时候，指甲下面全是干涸的血。」 |
| **推镜头** | 从远景缩到特写，制造压迫 | 「满朝文武跪了一地——没人敢抬头——龙椅上的手在发抖。」 |
| **拉镜头** | 从特写拉到远景，制造苍凉/宏大 | 「他松开手，那枚珠子滚落台阶，越滚越远，直到淹没在万家灯火里。」 |
| **跟拍** | 跟随角色运动，制造紧张 | 「三个拐角。左转。再左转。死胡同。」 |

**镜头使用规则**：
- 高潮场景 → 推镜头（先全景后特写，聚焦爆发瞬间）
- 转折场景 → 拉镜头（从微观拉到宏观，制造命运感）
- 追逐/战斗 → 跟拍+快速切换（短句、碎句模拟剪辑速度）
- 安静场景 → 允许留白（远景+风声+一句对白）
- 每个重要场景遵循「远→中→特」或「特→中→远」切换，不可全程中景平铺

#### 10.3.2 技法二：感官五重奏

每个重要场景至少激活3种感官，战斗/高潮场景5种全满。

| 感官 | 常见写法（❌） | 高级写法（✅） |
|------|--------------|--------------|
| **视觉** | 他看到了一片火海 | 「火光在他瞳孔里跳了跳。」 |
| **听觉** | 他听到了骨头断裂的声音 | 「骨头断裂的声音很轻，像踩断一根干树枝。」 |
| **触觉** | 他感到了寒意 | 「风从裂缝灌进来，贴着皮肤爬，像一千根手指。」 |
| **嗅觉** | 他闻到了血腥味 | 「空气里有一股甜味——他认出那是大量失血才有的气味。」 |
| **味觉** | 他尝到了铁锈味 | 「嘴里泛起铁锈味，舌头舔了舔——咬破了，但不是自己咬的。」 |

**铁律**：「他很害怕」「她很愤怒」「他感到悲伤」← 这是**告诉**，不是画面。用身体和感官说情绪。
- 害怕 → 「指甲抠进砖缝，后背贴墙，不敢呼吸。」
- 愤怒 → 「杯子碎了。他低头看了看手，松开，碎片叮叮当当落了一地。」
- 悲伤 → 「他站在那里，很久，雨水把肩膀打湿了一半才发现天在下雨。」

#### 10.3.3 技法三：动作微分解

关键动作拆成1-3秒的微动作，制造慢动作/子弹时间效果。

| 宏观写法（❌报告） | 微分解写法（✅电影） | 效果差异 |
|-------------------|-------------------|---------|
| 他挥剑斩杀了敌人 | 「剑从鞘中滑出——刀光一线——喉咙处绽开一道红线。」 | 前者是报告，后者是电影 |
| 她哭了 | 「嘴唇动了一下，鼻子皱了皱，然后眼泪才掉下来。」 | 前者是标签，后者是人 |
| 他一拳打飞了对手 | 「拳头碰上颧骨的瞬间，感觉到了骨头的让步——然后是空气——然后对手的身体在路上弹了两下。」 | 力量感完全不同 |
| 他接住了刀 | 「刀刃在他指缝间停住，血从指缝渗出来，顺着刀身往下淌。」 | 危险感和掌控感同时存在 |

**适用场景**：一招制敌 / 关键反转 / 角色首次展露实力 / 死亡瞬间 / 情感爆发
**不适用**：日常对话 / 走路等过渡动作（过度微分解 = 拖节奏）

#### 10.3.4 技法四：环境情绪投射

环境不是背景板，是情绪的放大器。场景的天气/光线/空间状态**必须服务当前叙事情绪**。

| 情绪 | 环境投射（✅画面） | 直述（❌告诉） |
|------|-------------------|--------------|
| 压迫 | 低矮天花板、昏暗灯光、逼仄空间 | 「他感到很大压力」 |
| 危机 | 暴风雨、闪电、裂缝、不稳定的结构 | 「情况很危险」 |
| 觉醒/突破 | 光线从缝隙涌入、天空放晴、冰面碎裂 | 「他突破了境界」 |
| 孤独 | 空旷、回声、单一光源 | 「他很孤独」 |
| 杀意 | 气温骤降、呼吸可见、万物死寂 | 「他杀气腾腾」 |
| 希望破灭 | 灯火渐次熄灭、声音远去、温度流失 | 「一切都没了」 |

**规则**：不用形容词说情绪，用环境说情绪。环境变化 = 情绪变化的视觉化。

#### 10.3.5 技法五：物件象征

关键物件 = 压缩的情感和叙事。一件物品可以承载伏笔、情感和隐喻。

**物件三阶段**：

| 阶段 | 目的 | 写法 |
|------|------|------|
| **1.首次出现** | 功能性引入，在行动中自然带出 | 「他从地上捡起那枚铜钱，掂了掂，揣进怀里。」 |
| **2.再次出现** | 叠加情感/象征，物件未变，含义已深 | 「铜钱在怀里硌着胸口，像那个人的目光还留在上面。」 |
| **3.最终出现** | 封印叙事使命，物件完成它的宿命 | 「铜钱滚落在坟前的泥地里，他没有捡。」 |

**关键物件判定**：与核心线索/核心关系绑定的物件 = 关键物件。每卷至少1-2件关键物件走完三阶段。

#### 10.3.6 技法六：声画对位

声音和画面不一致时，张力最大。

| 声音 | 画面 | 效果 |
|------|------|------|
| 轻柔的声音 | 屠杀现场 | 暴力的反差放大恐怖 |
| 嘈杂的人群 | 角色内心的寂静 | 孤独感的视觉化 |
| 一声尖叫 | 慢动作特写 | 时间凝固感 |
| 万物安静 | 暴风雨前 | 读者比角色更慌 |
| 欢声笑语 | 角色知道自己将死 | 悲哀的预知感 |

**写法**：「身后人声鼎沸，觥筹交错。他站在阳台上，只听得见自己的心跳。」

### 10.4 爽度+画面感联合规则

> 爽点必须有画面，画面必须服务爽点。两条铁律不可违反。

**铁律一：爽点必须有画面**
- 任何爽点（打脸/碾压/翻盘）至少使用2种画面感技法来呈现
- ❌ 「他一拳打倒了对手，全场震惊。」→ 这是**新闻**
- ✅ 「拳头落地的声音沉闷而短促。对面那个三阶修士的眼睛还瞪着——身体已经飞了出去。观战台安静了三秒。然后有人低声说：『他……他也是三阶？』」→ 这是**电影**

**铁律二：画面必须服务叙事目标**
- 环境描写/感官细节不能独立存在——如果一段画面描写删掉后爽感不减，那段画面是废画面
- 每一个环境细节都应该强化当前的叙事目标（压迫→更压迫，爆发→更震撼）

**冲突时优先级**：
```
爽度 > 画面感 > 文笔修饰
```
- 宁可糙一点但爽到，不可漂亮但无聊
- 画面感可以精简但不能没有（无画面的爽 = 干巴巴的战报）
- 文笔修饰最先砍（形容词/排比句/华丽辞藻在爽感面前一文不值）

### 10.5 每章爽度+画面感自检

> 与 §9.1 的叙事自检合并执行，每章完成后快速过一遍。

```
□ 【爽度密度】最近3章内是否有至少1个中爽点？5章内至少1个大爽点？
□ 【爽点构建】本章爽点是否走完三步（压制→转折→释放）？
□ 【旁观者】大爽点是否有2层以上旁观者反应？
□ 【爽点类型】近5章爽点类型是否有变化？（打脸→翻盘→碾压 轮换，不可连3次同类型）
□ 【镜头】本章是否有至少1个「远→中→特」或「特→中→远」镜头流程？
□ 【动作微分解】高潮/爽点的关键动作是否做了微分解？
□ 【感官】重要场景是否激活3种以上感官？高潮场景5种？
□ 【环境投射】环境描写是否服务当前情绪？删掉后情绪表达是否受损？（不受损=废画面）
□ 【告诉→画面】有无「他很X」式情感直述可替换为身体/环境画面？至少替换2处
□ 【物件】关键物件是否有重复出现并叠加含义？
```

| §9.2 断章控制 | 大高潮分拆法则 + 爽点三步法 = 高潮分拆时每段都是完整的小爽弧 |

## 关键陷阱

> 当前覆盖 15/70 种逻辑问题(21%)，完整分类见 `references/logic-consistency-taxonomy.md`

**逻辑类陷阱**：
- **死人复活**：status=dead 角色不能再有行动/对话
- **物品凭空出现/消失**：inventory 追踪归属，转移须有事件
- **距离/时间矛盾**：移动时间须与距离/方式匹配
- **伏笔烂尾**：major 线索长期 resolved_chapter=null 会持续告警
- **修为/能力膨胀**：增长须符合 power_system.rules 的进阶速度
- **角色分裂**：同一角色矛盾性格会被 key_traits 比对标记

**爽度类陷阱**（§10.2）：
- **假压制真装逼**：读者一眼看穿主角在演戏/藏实力→憋屈感不足→释放时无爽感。压制必须让读者也感到"完了"
- **天降翻盘**：无铺垫的外力/巧合拯救主角→作弊感→读者不爽反骂。转折必须有伏笔支撑
- **打脸拖沓**：碾压释放超过3段还没完成→爽感被稀释。释放要快准狠，2段内完成
- **爽点单调**：连续3次同类型爽点（如连3次打脸）→耐受性导致爽感递减。每5章至少覆盖3种类型
- **旁观者缺失**：爽点只有主角自己知道→爽感打折50%。大爽点至少2层旁观者反应

**画面感类陷阱**（§10.3）：
- **告诉代替画面**：「他很害怕」「她很愤怒」→ 用身体/环境/感官替换，每章至少替换2处
- **全程中景**：一个场景从头到尾都是同一距离视角→读者无法聚焦。重要场景「远→中→特」切换
- **废画面**：环境描写删掉后情绪表达不受影响→那段描写是废画面，删。画面必须服务叙事目标
- **微分解过度**：日常过渡动作也做微分解→拖节奏。微分解只用于关键时刻
- **感官单一**：只有视觉→扁平。重要场景至少3种感官，高潮5种全满
